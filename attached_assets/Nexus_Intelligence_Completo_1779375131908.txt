# 1. Visão geral

## 1.1 Conceito

Plataforma SaaS multi-tenant para empresas de semijoias que integra validação jurídica, gestão operacional, vendas, atendimento AI, documentação digital, notas fiscais, financeiro, contabilidade e marketing em um único app.

## 1.2 Objetivo

Oferecer uma solução completa que elimine a necessidade de múltiplos sistemas, automatizando processos desde o marketing até a contabilidade.

## 1.3 Diferenciais competitivos

1. Única plataforma que integra validação jurídica + gestão + vendas + fiscal + contábil
2. Automação end-to-end
3. IA integrada em múltiplos pontos
4. Multi-canal (WhatsApp, e-mail, calendário, app)
5. Arquitetura escalável (multi-tenant)
6. Específico para semijoias

# 2. O que já existe e funciona

## 2.1 Sistema de atendimento AI (n8n)

### 2.1.1 Atendente AI "Amanda"

**Funcionalidades:**
- Agente conversacional para Dra. Ana Miranda (terapia)
- Memória persistente (Postgres Chat Memory + Redis)
- Context window configurável (20 mensagens)
- Session ID baseado em telefone
- Controle de sessão (60 min de inatividade)

**Tools integradas:**
- `enviar_formulario` - Envia formulário de qualificação
- `criar_reuniao` - Agenda reunião no Google Calendar
- `reagendar_reuniao` - Reagenda reunião existente
- `cancelar_reuniao` - Cancela reunião
- `cadastrar_equipe` - Cadastra funcionários da empresa
- `enviar_img1` - Envia PDF/ficha via WhatsApp
- `Cerebro` - Tool de FAQ e conhecimento geral

**Sistema de pausa/reativação:**
- Campo `atendimento_ia` em `dados_cliente`
- Valores: "pause" (humano assume), "reativada" (IA volta)
- Switch para rotear mensagens
- Histórico salvo mesmo quando pausada

**Divisão inteligente de mensagens:**
- Split de mensagens longas em partes
- Delay de 1 segundo entre envios
- Loop para enviar todas as partes
- Evita bloqueio do WhatsApp

**Integrações:**
- Evolution API (WhatsApp)
- Google Calendar (agendamento)
- Google Gemini (transcrição)
- OpenAI (atendente AI)
- Supabase (banco de dados)
- Redis (cache)

### 2.1.2 Fluxo de atendimento

```
Webhook EVO1 → Dados1 → Rotas1 → Switch8 (incoming/outcoming)
  ↓
Supabase1 (verifica cliente) → If2 (existe?)
  ↓
Switch7 (tipo: text/audio/image) → Processamento
  ↓
Get Memory (Redis) → Mensagem Completa1
  ↓
Switch (IA ativa/pausada) → Amanda (AI) ou Humano
  ↓
Salvar histórico → Enviar resposta
```

# 3. O que precisa ser construído

## 3.1 Módulo jurídico (Datacorp)

### 3.1.1 Funcionalidades

**Consulta de CPF:**
- Processos judiciais vinculados ao CPF
- Processos financeiros (pendências, restrições)
- Validação de dados pessoais:
  - CPF válido?
  - Nome confere?
  - Endereço correto?
  - Telefone válido?
  - E-mail válido?

**Score de confiabilidade:**
- Sistema de pontuação (0-100 pontos)
- Critérios:
  - Dados pessoais válidos: 0-20 pontos
  - Sem processos judiciais: 0-30 pontos
  - Sem pendências financeiras: 0-30 pontos
  - Score de crédito: 0-20 pontos

**Classificação:**
- 80-100 pontos: Aprovação automática
- 50-79 pontos: Análise manual
- 0-49 pontos: Reprovação automática

**Monitoramento contínuo:**
- Alertas automáticos sobre mudanças
- Reavaliação periódica
- Histórico de scores

**Integração:**
- API Datacorp (BigDataCorp)
- TOKEN_ID e CHAVE_TOKEN configurados
- Validação no cadastro de revendedoras

### 3.1.2 Fluxo de validação

```
1. Revendedora preenche formulário
   ↓
2. Sistema envia dados para Datacorp
   ├─ Validação de Dados Pessoais
   ├─ Consulta Processos Judiciais
   └─ Consulta Processos Financeiros
   ↓
3. Sistema calcula Score de Confiabilidade
   ↓
4. Decisão Automática
   ├─ Score ≥ 80: Aprovação Automática
   ├─ Score 50-79: Análise Manual
   └─ Score < 50: Reprovação Automática
   ↓
5. Notificação e Próximos Passos
```

## 3.8 Contabilidade e fiscal automatizada

### 3.8.1 Funcionalidades

**Geração automática de material para contador:**

**A) DRE (Demonstração do Resultado do Exercício)**
```
RECEITAS
├─ Receita Bruta (todas as vendas)
├─ (-) Impostos sobre vendas (ICMS, PIS, COFINS)
├─ (-) Devoluções e cancelamentos
└─ = Receita Líquida

CUSTOS
├─ Custo das Mercadorias Vendidas (CMV)
├─ Comissões pagas às revendedoras
└─ = Lucro Bruto

DESPESAS OPERACIONAIS
├─ Despesas Administrativas
├─ Despesas de Vendas
└─ = Lucro Operacional

RESULTADO FINANCEIRO
├─ Receitas financeiras
├─ (-) Despesas financeiras
└─ = Lucro Antes do IR

IMPOSTO DE RENDA
├─ IRPJ (Imposto de Renda Pessoa Jurídica)
├─ CSLL (Contribuição Social sobre Lucro Líquido)
└─ = Lucro Líquido
```

**B) Balanço Patrimonial**
```
ATIVO
├─ Circulante (Caixa, Contas a receber, Estoque)
└─ Não Circulante (Imobilizado)

PASSIVO
├─ Circulante (Fornecedores, Impostos, Empréstimos)
└─ Não Circulante

PATRIMÔNIO LÍQUIDO
├─ Capital Social
└─ Lucros Acumulados
```

**C) Livros Fiscais**
- Livro de Entradas (compras)
- Livro de Saídas (vendas)
- Apuração de ICMS
- Apuração de PIS/COFINS
- Controle de NF-e

**D) Relatórios por Revendedora**
- Faturamento individual
- Comissões pagas
- Impostos retidos na fonte (se aplicável)
- Relatório para emissão de recibo

**E) Conciliação Bancária**
- Extrato bancário vs. movimentações do sistema
- Identificação de diferenças
- Saldo ajustado

### 3.8.2 Cálculo automático de impostos

**IRPJ (Imposto de Renda Pessoa Jurídica):**
```
Base de Cálculo = Lucro Real ou Presumido
├─ Se Lucro Real:
│   └─ IRPJ = 15% sobre lucro + 10% sobre lucro > R$ 20.000/mês
│
└─ Se Lucro Presumido:
    └─ IRPJ = 15% sobre presunção (ex: 8% da receita)
```

**CSLL (Contribuição Social sobre Lucro Líquido):**
```
CSLL = 9% sobre lucro (lucro real)
ou
CSLL = 12% sobre presunção (lucro presumido)
```

**ICMS (Imposto sobre Circulação de Mercadorias):**
```
ICMS = Alíquota estadual × Valor da venda
├─ Calculado por estado
├─ Créditos de ICMS (compras)
└─ ICMS a recolher = Débito - Crédito
```

**PIS/COFINS:**
```
PIS = 0,65% sobre receita (regime cumulativo)
COFINS = 3% sobre receita (regime cumulativo)
ou
PIS = 1,65% sobre receita (regime não cumulativo)
COFINS = 7,6% sobre receita (regime não cumulativo)
```

### 3.8.3 Sistema de pagamento automático de impostos

**Fluxo:**
```
1. CÁLCULO AUTOMÁTICO
   ├─ Sistema calcula impostos do mês
   ├─ Valida com regras fiscais
   └─ Gera DARF/GPS
   ↓
2. REVISÃO E APROVAÇÃO
   ├─ Contador revisa cálculos
   ├─ Aprova ou ajusta valores
   └─ Sistema gera documento final
   ↓
3. GERAÇÃO DE DOCUMENTOS
   ├─ DARF (IRPJ, CSLL)
   ├─ GPS (INSS, se aplicável)
   └─ GARE (ICMS, se estadual)
   ↓
4. INTEGRAÇÃO BANCÁRIA
   ├─ Conecta com banco
   ├─ Cria ordem de pagamento
   ├─ Agenda para data de vencimento
   └─ Executa pagamento automaticamente
   ↓
5. COMPROVAÇÃO
   ├─ Baixa comprovante de pagamento
   ├─ Atualiza status no sistema
   └─ Arquiva documentos
```

### 3.8.4 Tabelas necessárias

**`impostos_calculados`:**
- `id`, `periodo` (mês/ano)

# Módulo de Envios e Logística - Documentação Completa

## 1. Visão geral

### 1.1 Objetivo
Sistema de gestão de envios integrado que permite calcular fretes, gerar etiquetas, rastrear entregas e gerenciar logística de forma automatizada, servindo como plataforma intermediária de envios.

### 1.2 Escopo
- Cálculo automático de frete
- Comparação de múltiplas transportadoras
- Geração automática de etiquetas
- Rastreamento unificado
- Gestão de envios
- Multi-tenant (SaaS)
- Suporte a múltiplas transportadoras

---

## 2. Arquitetura do sistema

### 2.1 Tecnologias
- Backend: Node.js (Replit)
- Banco de dados: Supabase (PostgreSQL)
- Cache: Redis (opcional)
- APIs externas:
  - Melhor Envio (inicial)
  - Correios API
  - Jadlog API
  - Azul Cargo API
  - Loggi API
  - LATAM Cargo API
  - Outras transportadoras

### 2.2 Arquitetura multi-tenant
- Código único para todos os clientes
- Isolamento por credenciais/configurações
- Cada cliente usa suas próprias credenciais de transportadora
- Configuração centralizada no banco de dados

---

## 3. Estratégia de implementação

### 3.1 Fase 1: Inicial (Melhor Envio)
Objetivo: Começar rapidamente sem burocracia

Vantagens:
- API gratuita
- Sem volume mínimo
- Sem burocracia
- Múltiplas transportadoras integradas
- Documentação completa
- Ambiente de testes (Sandbox)

Implementação:
- Integrar API do Melhor Envio
- Configurar tokens por cliente
- Implementar cálculo de frete
- Implementar geração de etiquetas
- Implementar rastreamento

Custos:
- API: gratuita
- Paga apenas: frete quando envia (com margem deles 30-40%)

### 3.2 Fase 2: Migração (Transportadoras diretas)
Objetivo: Reduzir custos quando tiver volume

Condição: ~4 envios/semana (16-20/mês)

Transportadoras priorizadas:
1. Loggi (1-3 dias para aprovação)
   - Volume mínimo: baixo/sem mínimo
   - Facilidade: alta

2. Azul Cargo (3-7 dias)
   - Volume mínimo: 30-100 envios/mês
   - Facilidade: alta

3. Jadlog (5-10 dias)
   - Volume mínimo: 50-200 envios/mês
   - Facilidade: média

4. LATAM Cargo (7-14 dias)
   - Volume mínimo: 100-300 envios/mês
   - Facilidade: média

5. Correios (7-15 dias)
   - Volume mínimo: 100-500 envios/mês
   - Facilidade: baixa (mais burocrático)

Benefícios da migração:
- Economia: 30-40% (sem margem do Melhor Envio)
- Maior controle
- Descontos progressivos com volume
- Integração direta

## 6. Fluxos de trabalho

### 6.1 Fluxo: Cálculo de Frete

```
1. Cliente adiciona produto no carrinho
   ↓
2. App chama: POST /api/envio/:clienteId/calcular-frete
   ↓
3. Backend busca credenciais do cliente no banco
   ↓
4. Backend identifica transportadora ativa
   ↓
5. Backend chama API da transportadora (Melhor Envio/Loggi/etc)
   ↓
6. Backend recebe cotações
   ↓
7. Backend processa e normaliza respostas
   ↓
8. Backend salva cotações no banco (cache)
   ↓
9. Backend retorna opções para o app
   ↓
10. App exibe opções para cliente escolher
```

Tempo estimado: 2-5 segundos

### 6.2 Fluxo: Geração de Etiqueta (Após Pagamento)

```
1. Cliente paga pedido (PIX/Cartão)
   ↓
2. Pagamento confirmado (webhook gateway)
   ↓
3. App chama: POST /api/envio/:clienteId/gerar-etiqueta
   ↓
4. Backend valida pagamento do pedido
   ↓
5. Backend busca credenciais do cliente
   ↓
6. Backend chama API transportadora para gerar etiqueta
   ↓
7. Backend recebe PDF da etiqueta
   ↓
8. Backend salva PDF no Supabase Storage/Google Drive
   ↓
9. Backend salva informações do envio no banco
   ↓
10. Backend envia notificação WhatsApp para revendedora
    (via Evolution API)
   ↓
11. Backend retorna resposta para app
```

Tempo estimado: 10-20 segundos (PIX) ou 30-50 segundos (Cartão)

### 6.3 Fluxo: Rastreamento Automático

```
1. Envio criado no sistema
   ↓
2. Cron job executa a cada 1 hora (ou webhook)
   ↓
3. Backend busca envios pendentes/em trânsito
   ↓
4. Para cada envio:
   a. Backend chama API transportadora para rastrear
   b. Backend recebe atualizações
   c. Backend compara com último status conhecido
   d. Se houver nova atualização:
      - Salva evento no banco
      - Atualiza status do envio
      - Notifica cliente via WhatsApp
      - Notifica revendedora via WhatsApp
   ↓
5. Repete até envio entregue
```

### 6.4 Fluxo: Notificações de Status

```
Status: "etiqueta_gerada"
   ↓
Notifica revendedora: "Etiqueta pronta para impressão"
   ↓
Status: "postado"
   ↓
Notifica cliente: "Seu pedido foi postado. Código: XXX"
   ↓
Status: "em_transito"
   ↓
Notifica cliente: "Seu pedido está a caminho"
   ↓
Status: "saiu_para_entrega"
   ↓
Notifica cliente: "Seu pedido saiu para entrega hoje"
   ↓
Status: "entregue"
   ↓
Notifica cliente: "Pedido entregue com sucesso!"
Notifica revendedora: "Pedido entregue ao cliente"
```

// services/transportadoras.service.js
class TransportadoraService {
  constructor(clienteId) {

      'loggi': LoggiAPI,

  }

  async calcularFrete(dados) {
    const transportadora = await this.getTransportadoraAtiva();
    const provider = await this.getProvider(transportadora);

    const transportadora = await this.getTransportadoraAtiva();
    const provider = await this.getProvider(transportadora);

  }

## 7. Implementação Técnica

### 7.1 Arquitetura Híbrida

### 7.2 Detecção Automática de Modo

#### Estados do Sistema:

```
Estado: "APRESENTACAO"
- Avatar reproduz vídeo gravado
- Custo: R$ 0,00
- Duração: ~40 minutos

Estado: "AGUARDANDO_PERGUNTA"
- Monitora áudio (nível mínimo)
- Custo: ~R$ 0,01
- Duração: variável

Estado: "PROCESSANDO_PERGUNTA"
- Speech-to-Text + Gemini + TTS
- Custo: ~R$ 0,21 por pergunta
- Duração: ~2-5 segundos
```

#### Implementação:

```javascript
class ReuniaoFlow {
  async iniciar() {
    // 1. Modo Apresentação
    this.modo = 'APRESENTACAO';
    await this.reproduzirApresentacao();
    
    // 2. Modo Aguardando
    this.modo = 'AGUARDANDO_PERGUNTA';
    await this.aguardarPergunta();
  }
  
  async reproduzirApresentacao() {
    // Vídeo pré-gravado - custo zero
    await avatar.reproduzirVideo('apresentacao_empresa.mp4');
  }
  
  async aguardarPergunta() {
    while (!this.temPergunta()) {
      await this.verificarAudio(); // Custo mínimo
    }
    
    // Cliente fez pergunta → Processar
    this.modo = 'PROCESSANDO_PERGUNTA';
    await this.processarPergunta();
    
    // Volta para aguardar
    this.modo = 'AGUARDANDO_PERGUNTA';
    await this.aguardarPergunta();
  }
  
  async processarPergunta() {
    // Custo REAL aqui
    const audio = await capturarAudio();
    const texto = await speechToText(audio); // R$ 0,06
    const resposta = await gemini.processar(texto); // R$ 0,064
    const audioResposta = await textToSpeech(resposta); // R$ 0,10
    await avatar.falar(audioResposta); // R$ 0,30
  }
}
```
