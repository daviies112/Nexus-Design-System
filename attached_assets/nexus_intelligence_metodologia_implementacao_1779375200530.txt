================================================================================
VISÃO GERAL DA METODOLOGIA
================================================================================

A metodologia Nexus é dividida em 5 grandes etapas que cobrem toda a jornada
do cliente: desde o primeiro contato comercial até a operação consolidada e
eventual expansão de plano.

Cada etapa tem fases internas, responsáveis definidos, prazos, checklists de
conclusão e critérios de avanço. Nenhuma etapa começa antes da anterior ser
concluída e validada.

ETAPA 0 — PRÉ-VENDA E DIAGNÓSTICO
ETAPA 1 — ONBOARDING TÉCNICO (dias 1 a 7)
ETAPA 2 — ONBOARDING OPERACIONAL (dias 8 a 14)
ETAPA 3 — OPERAÇÃO CONTÍNUA (mês a mês)
ETAPA 4 — EXPANSÃO DE PLANO
ETAPA 5 — OFFBOARDING

Responsáveis internos da Nexus:
  CS (Customer Success): acompanhamento do cliente, treinamento, suporte.
  TEC (Técnico): configuração de infraestrutura, workflows, integrações.
  COM (Comercial): gestão do relacionamento de conta e expansão.

Prazos são contados em dias úteis a partir da data de assinatura do contrato.


================================================================================
ETAPA 0 — PRÉ-VENDA E DIAGNÓSTICO
================================================================================
Responsável: COM + CS
Duração: variável (durante o processo comercial)
Objetivo: entender a operação atual do cliente antes de vender, para
configurar corretamente e definir o plano certo desde o início.


FASE 0.1 — FORMULÁRIO DE DIAGNÓSTICO PRÉ-VENDA

Antes de apresentar proposta, o comercial aplica o diagnóstico com o prospect.
Esse diagnóstico define qual plano recomendar e quais integrações ativar.

Perguntas obrigatórias do diagnóstico:

  OPERAÇÃO ATUAL
  1. Quantas revendedoras ativas você tem hoje?
  2. Quantos leads novos você recebe por mês (estimativa)?
  3. Como você prospecta hoje? (indicação, redes sociais, anúncios, outros)
  4. Você usa WhatsApp para prospecção? É um número pessoal ou comercial?
  5. Como você faz a qualificação das revendedoras hoje?
  6. Você consulta CPF antes de contratar? Como?
  7. Você tem contrato digital hoje? Como assina?

  GESTÃO FINANCEIRA
  8. Como você cobra o repasse das revendedoras hoje?
  9. Quantas revendedoras inadimplentes você tem em média por mês?
  10. Você já negativou alguém no Serasa? Como foi o processo?
  11. Você emite NF-e para as revendedoras? Via qual sistema?

  LOGÍSTICA
  12. Você trabalha com maletas consignadas?
  13. Como você controla o que cada revendedora está com você?
  14. Você usa Melhor Envio ou transportadora própria?
  15. Como você gerencia devoluções por defeito (RMA)?

  FERRAMENTAS ATUAIS
  16. Quais ferramentas você usa hoje? (Jueri, Up Vendas, outros)
  17. Quanto paga por mês em ferramentas?
  18. O que mais te incomoda na operação atual?

  METAS
  19. Quantas revendedoras você quer ter em 6 meses?
  20. Qual o seu maior gargalo hoje para crescer?

CRITÉRIO DE CONCLUSÃO DA ETAPA 0:
  [ ] Diagnóstico preenchido e registrado no CRM interno da Nexus
  [ ] Plano recomendado definido com justificativa
  [ ] Integrações necessárias mapeadas (Bling, Efi Bank, Melhor Envio, Serasa)
  [ ] Proposta enviada com o plano correto
  [ ] Contrato assinado e pagamento do primeiro mês confirmado

ETAPA 1 — ONBOARDING TÉCNICO
================================================================================
Responsável principal: TEC
Responsável de acompanhamento: CS
Duração: dias 1 a 7 após assinatura
Objetivo: ter toda a infraestrutura técnica configurada, testada e funcionando
antes de qualquer treinamento com o cliente.


FASE 1.1 — PROVISIONAMENTO DO TENANT (dia 1)
Responsável: TEC

  BANCO DE DADOS E INFRAESTRUTURA
  [ ] Criar schema exclusivo no PostgreSQL para o tenant
  [ ] Configurar namespace no Redis para o tenant
      (padrão: instance_name + ':' + telefone como chave de sessão)
  [ ] Registrar tenant_id no sistema multi-tenant
  [ ] Configurar limites de plano no middleware de autorização:
      Start: 30 leads únicos/mês, sem Pix automático, sem maleta auto,
             sem follow-up inadimplência, sem anti-churn, sem NF-e,
             sem RMA IA, sem negativação, sem plano de ação IA
      Pro:   70 leads únicos/mês, revendedoras ilimitadas, todas as
             automações financeiras e logísticas ativas
      Max:   ilimitado em tudo, negativação e plano de ação IA ativos
  [ ] Registrar plano ativo, data de início e data de renovação
  [ ] Configurar billing: cartão cadastrado, dia 1 mensalidade,
      dia 5 uso avulso
  [ ] Confirmar cobrança do primeiro mês processada com sucesso

  WHATSAPP — EVOLUTION API
  [ ] Provisionar instância exclusiva no Evolution API para o tenant
  [ ] Conectar o número de WhatsApp do cliente (QR code ou pareamento)
  [ ] Confirmar status connected no Evolution API
  [ ] Testar envio e recebimento de mensagem de texto
  [ ] Testar envio e recebimento de áudio
  [ ] Configurar webhook de recebimento apontando para o n8n do tenant
  [ ] Registrar número da instância no sistema como tenant_id principal

  ATIVAÇÃO DO WORKFLOW PRINCIPAL (n8n)
  [ ] Importar workflow base Emerick para o tenant
  [ ] Configurar variáveis de ambiente do tenant:
      - INSTANCE_NAME (nome da instância WhatsApp)
      - TENANT_ID
      - POSTGRES_URL (schema do tenant)
      - REDIS_PREFIX (prefixo exclusivo do tenant)
      - OPENAI_API_KEY (chave da Nexus, compartilhada)
      - GEMINI_API_KEY (chave da Nexus, compartilhada)
      - DATACORP_KEY (para consulta de CPF)
  [ ] Ativar cron de monitoramento de contratos (a cada 1 minuto)
  [ ] Ativar cron de follow-up de formulários (a cada 3 minutos)
  [ ] Ativar cron de follow-up de contratos (a cada 5 minutos)
  [ ] Confirmar que os crons estão rodando no n8n sem erro
  [ ] Ativar cron diário 8h: reset de estoque
  [ ] Ativar cron diário 9h: ciclo financeiro (somente Pro/Max)
  [ ] Ativar cron diário 10h30: anti-churn (somente Pro/Max)
  [ ] Ativar cron semanal 2ª 7h: NF-e em lote (somente Pro/Max)
  [ ] Ativar cron dia 1 de cada mês 8h: ciclo mensal

  CHECKLIST DE VALIDAÇÃO TÉCNICA DO DIA 1:
  [ ] Banco criado e acessível
  [ ] Redis funcionando com prefixo correto
  [ ] WhatsApp conectado e respondendo
  [ ] Workflow ativo sem erros no n8n
  [ ] Todos os crons do plano ativos e sem erro

  Duração: 30 minutos

  Pauta:
  [ ] Confirmar que todos os módulos funcionam corretamente
  [ ] Confirmar que o cliente se sente seguro para operar
  [ ] Entregar lista de suporte: canal WhatsApp, e-mail, SLA de resposta
  [ ] Apresentar agenda de check-ins dos primeiros 90 dias
  [ ] Coletar NPS do onboarding (de 0 a 10, qual a chance de indicar?)
  [ ] Declarar cliente "operacional"

  CRITÉRIO DE CONCLUSÃO DA ETAPA 2:
  [ ] Simulação completa executada com sucesso pelo cliente sem ajuda
  [ ] NPS coletado
  [ ] Cliente declarado operacional
  [ ] Registro de go-live no sistema interno com data e responsável


================================================================================
ETAPA 3 — OPERAÇÃO CONTÍNUA
================================================================================
Responsável: CS
Objetivo: garantir que o cliente está usando a plataforma, crescendo e não
abandonando. Prevenir churn antes que apareça.


ROTINA DE CHECK-IN — PRIMEIROS 90 DIAS (período crítico)

  CHECK-IN DE 7 DIAS (D+7 do go-live)
  Responsável: CS
  Formato: WhatsApp ou ligação de 15 minutos
  Verificar:
  [ ] Quantos leads entraram desde o go-live?
  [ ] Algum erro ou comportamento inesperado da Amanda?
  [ ] Alguma dificuldade em algum módulo?
  [ ] O cliente está usando o CRM?
  [ ] Alguma integração deu problema?
  Ação: resolver qualquer problema imediatamente. Registrar no sistema.

  CHECK-IN DE 30 DIAS (D+30 do go-live)
  Responsável: CS
  Formato: reunião de 30 minutos via Google Meet
  Verificar:
  [ ] Quantos leads qualificados no mês?
  [ ] Quantos contratos assinados?
  [ ] O cliente chegou perto do limite de leads do plano?
  [ ] Qual foi o custo de uso avulso do mês?
  [ ] O cliente entendeu a fatura do dia 5?
  [ ] Alguma funcionalidade não está sendo usada que deveria?
  [ ] O cliente tem alguma revendedora inadimplente que o sistema está
      acompanhando? (Pro/Max)
  Ação:
  - Se próximo do limite de leads → iniciar conversa de upgrade
  - Se não usando funcionalidades → reforçar treinamento
  - Se insatisfeito → escalar para CS sênior imediatamente
  Coletar NPS D+30.

  CHECK-IN DE 60 DIAS (D+60 do go-live)
  Responsável: CS
  Formato: reunião de 30 minutos
  Verificar:
  [ ] Crescimento de revendedoras ativas comparado ao mês anterior
  [ ] Taxa de inadimplência e recuperação automática (Pro/Max)
  [ ] Uso de marketing e gamificação
  [ ] Custo por revendedora ativa no mês
  [ ] Alguma funcionalidade do plano superior que o cliente está
      precisando mas não tem?
  Ação:
  - Apresentar comparativo de ROI: o que gastou vs o que gerou
  - Se sinal de upgrade → passar para o COM

  CHECK-IN DE 90 DIAS (D+90 do go-live)
  Responsável: CS + COM
  Formato: reunião de 45 minutos — revisão de conta
  Verificar:
  [ ] Revisão completa dos 3 meses: leads, contratos, revendedoras,
      inadimplência, recuperação, custo total, ROI
  [ ] NPS D+90
  [ ] Metas do cliente: onde estava, onde está, onde quer estar
  [ ] Funcionalidades mais usadas e menos usadas
  [ ] O plano atual ainda é o correto?
  Ação:
  - Apresentar proposta de expansão se aplicável
  - Renovar compromisso com o cliente
  - Definir próxima revisão de conta (a cada 3 meses)


ROTINA MENSAL CONTÍNUA (após os 90 dias)

  TODO DIA 2 DE CADA MÊS — CS verifica automaticamente:
  [ ] Quais clientes chegaram em 80%+ do limite de leads no mês anterior?
      → Acionar COM para conversa de upgrade
  [ ] Quais clientes tiveram redução de leads em relação ao mês anterior?
      → Risco de churn — acionar CS para check-in de saúde
  [ ] Quais clientes não fizeram login na plataforma nos últimos 15 dias?
      → Risco de churn — contato imediato

  TODO DIA 6 DE CADA MÊS — CS verifica cobranças:
  [ ] Cartões com falha no dia 1 (mensalidade)?
      → Acionar cliente imediatamente para atualizar cartão
      → Plataforma em modo de aviso (não bloqueio imediato): 5 dias de
         carência antes de suspender
  [ ] Cartões com falha no dia 5 (uso avulso)?
      → Mesmo procedimento
  [ ] Clientes com uso avulso acima do threshold com alarme disparado?
      → Verificar se o COM já abordou o upgrade

  REVISÃO DE CONTA TRIMESTRAL (a cada 3 meses após os 90 dias)
  Responsável: CS + COM
  Formato: reunião de 45 minutos
  Pauta:
  [ ] Revisão de métricas do trimestre
  [ ] Comparativo de ROI
  [ ] Novas funcionalidades lançadas que o cliente ainda não usa
  [ ] Oportunidade de upgrade
  [ ] NPS trimestral


PROTOCOLO DE SUPORTE CONTÍNUO

  Canal de suporte: WhatsApp dedicado da Nexus + e-mail
  SLA de resposta por prioridade:
    Crítico (sistema fora do ar, Amanda não respondendo, Pix não
             processando): resposta em até 2 horas, resolução em até 4 horas
    Alto (integração com erro, cron não executou, contrato não gerou):
         resposta em até 4 horas, resolução no mesmo dia
    Médio (dúvida operacional, configuração, treinamento adicional):
          resposta em até 24 horas
    Baixo (sugestão de melhoria, dúvida não urgente):
          resposta em até 48 horas

  O cliente abre suporte via WhatsApp ou e-mail. O CS registra no sistema
  interno com categoria, prioridade e status. Nenhum chamado pode ficar sem
  atualização por mais de 4 horas em dias úteis.


================================================================================
ETAPA 4 — EXPANSÃO DE PLANO (UPGRADE)
================================================================================
Responsável: COM (gatilho) + CS (execução técnica)
Objetivo: identificar o momento certo de upgrade e executar sem atrito.


SINAIS DE GATILHO DE UPGRADE

  Start → Pro:
  - Cliente atingiu 80%+ do limite de leads em 2 meses seguidos
  - Cliente tem revendedoras ativas e está cobrando repasse manualmente
  - Cliente perguntou sobre Pix automático ou maleta automatizada
  - NPS abaixo de 7 com menção a falta de automação financeira

  Pro → Max:
  - Cliente tem histórico de revendedoras que chegam no D+15 sem pagar
  - Cliente perguntou sobre negativação Serasa
  - Volume de inadimplência do cliente está crescendo mês a mês
  - Cliente tem mais de 100 revendedoras ativas


PROCESSO DE UPGRADE

  PASSO 1 — ABORDAGEM COMERCIAL (COM)
  [ ] COM agenda reunião de 30 minutos com o decisor
  [ ] Apresenta comparativo: plano atual vs plano superior
  [ ] Apresenta cálculo de ROI:
      Ex: "Você tem 8 revendedoras inadimplentes por mês. Com o Max,
      a negativação automática no D+17 custaria R$ 200 por mês.
      Se recuperar mesmo que 2, já pagou. As 6 restantes param de
      reincidir após a primeira negativação."
  [ ] Mostra a tabela de funcionalidades bloqueadas que o cliente
      já viu na plataforma com o cadeado
  [ ] Propõe upgrade e aguarda decisão

  PASSO 2 — EXECUÇÃO TÉCNICA (TEC) — execução no mesmo dia da aprovação
  [ ] Atualizar plano do tenant no banco de dados
  [ ] Atualizar middleware de autorização (liberar funcionalidades do
      novo plano imediatamente)
  [ ] Ativar crons adicionais do novo plano
  [ ] Configurar novas integrações se necessário (ex: Serasa para Max)
  [ ] Calcular diferença proporcional da mensalidade (pro rata)
  [ ] Confirmar nova mensalidade no próximo dia 1

  PASSO 3 — TREINAMENTO DO MÓDULO NOVO (CS) — dentro de 48 horas
  [ ] Sessão de treinamento de 60 minutos focada apenas no que é novo
  [ ] Exercício prático com o cliente
  [ ] Confirmar que o cliente sabe usar as novas funcionalidades
  [ ] Enviar gravação

  CRITÉRIO DE CONCLUSÃO DO UPGRADE:
  [ ] Plano atualizado no sistema
  [ ] Funcionalidades disponíveis imediatamente
  [ ] Crons ativos
  [ ] Cliente treinado nas novas funcionalidades
  [ ] Registro de upgrade no sistema com data e responsável


================================================================================
ETAPA 5 — OFFBOARDING
================================================================================
Responsável: COM (retenção) + CS (execução) + TEC (técnico)
Objetivo: tentar reter o cliente, e se não for possível, encerrar de forma
organizada e preservar o relacionamento.


FASE 5.1 — DETECÇÃO E RETENÇÃO

  Sinais de pedido de cancelamento ou risco de churn:
  - Cliente solicita cancelamento diretamente
  - NPS abaixo de 5
  - Cliente não faz login há mais de 30 dias
  - Cliente com cobranças recusadas e sem atualização de cartão após 7 dias

  PROTOCOLO DE RETENÇÃO (antes de qualquer ação técnica):
  [ ] COM entra em contato imediatamente (máximo 24 horas após o sinal)
  [ ] Reunião de 30 minutos para entender o motivo real do cancelamento
  [ ] Tentativa de resolução:
      - Problema técnico: CS resolve imediatamente
      - Preço: avaliar possibilidade de downgrade em vez de cancelamento
      - Não está usando: CS agenda sessão de reativação e treinamento
      - Migrou para concorrente: documentar o motivo para produto
  [ ] Se retido: registrar no sistema com plano de acompanhamento reforçado
  [ ] Se não retido: seguir para fase 5.2


FASE 5.2 — PROCESSO DE ENCERRAMENTO

  COMUNICAÇÃO
  [ ] COM confirma decisão de encerramento por escrito (e-mail)
  [ ] CS informa ao cliente o que acontece com os dados após o encerramento
  [ ] CS informa o prazo de acesso à plataforma após o pedido
      (padrão: acesso até o final do ciclo pago)

  EXPORTAÇÃO DE DADOS (obrigatória — direito do cliente)
  [ ] CS disponibiliza exportação completa dos dados do cliente:
      - Todas as revendedoras cadastradas (nome, CPF, contato, status)
      - Histórico de contratos assinados (PDF)
      - Histórico financeiro (repasses, inadimplência, NF-es)
      - Histórico de leads do CRM
      - Catálogo de produtos e estoque
      - Histórico de conversas da Amanda (log)
  [ ] Exportação enviada ao cliente em formato .csv e .pdf compactados
  [ ] Cliente confirma recebimento dos dados por escrito

  DESCOMISSIONAMENTO TÉCNICO (TEC) — executado após fim do ciclo pago
  [ ] Desativar todos os crons do tenant no n8n
  [ ] Desativar workflow do tenant
  [ ] Desconectar instância do Evolution API
  [ ] Revogar tokens de todas as integrações (Google, Efi Bank, Bling,
      Melhor Envio, Serasa, redes sociais)
  [ ] Manter dados no PostgreSQL por 90 dias (obrigação legal) e depois
      deletar schema do tenant
  [ ] Remover sessões do Redis
  [ ] Remover registro do tenant do sistema de billing
  [ ] Confirmar que o cartão do cliente não receberá nenhuma cobrança
      após o encerramento

  PESQUISA DE SAÍDA
  [ ] CS envia pesquisa de saída com 3 perguntas:
      1. Por que você cancelou? (múltipla escolha + campo aberto)
      2. O que poderíamos ter feito diferente?
      3. Você indicaria a Nexus para outro negócio mesmo tendo cancelado?
  [ ] Respostas registradas no sistema de produto para melhoria contínua

  CRITÉRIO DE CONCLUSÃO DO OFFBOARDING:
  [ ] Dados exportados e entregues ao cliente
  [ ] Todos os serviços técnicos descomissionados
  [ ] Faturamento encerrado sem cobrança indevida
  [ ] Pesquisa de saída enviada
  [ ] Registro de encerramento no sistema com data, motivo e responsável
